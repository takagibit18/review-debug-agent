# Code Review & Debug Agent — 开源项目规划（意图对齐 · 架构 · 流程 · 分工）

> 面向：大三 CS，求职 **Agent 开发 / LLM 应用工程师**，2 人远程协作；顺带练习 **Git、Docker、CI、API 服务化**。  
> 状态：初稿，用于团队对齐；标「待确认」项需在首次同步会议拍板。

---

## 1. 意图对齐（我们到底在做什么）

### 1.1 背景与动机


| 维度    | 共识假设（可改）                                              |
| ----- | ----------------------------------------------------- |
| 简历故事  | 「做了一个可部署的 LLM Agent：能对本地/PR 代码做结构化 Review，并辅助定位与修复问题」 |
| 技术证明点 | Tool Calling / 多步推理、RAG（可选）、异步服务、评测、容器化、协作工程规范        |
| 协作目标  | 强制走 PR、Issue、分支；用 Docker 统一运行环境，降低「我机器能跑」问题           |


### 1.2 产品形态（当前推荐：先窄后宽）

**MVP 建议**：以 **仅 CLI** 为主（不做 GitHub PR 评论机器人），而不是一上来就做完整 IDE 插件或服务化平台。

- **输入**：本地仓库路径、或 Git diff / PR patch、或指定文件 + 错误日志/测试失败输出。  
- **输出**：  
  - **Review**：按严重级别分类的问题列表（安全/正确性/风格/可维护性）、可引用行号或 diff hunk。  
  - **Debug**：假设 → 验证步骤 → 建议补丁（或最小 diff），必要时「建议运行的命令」由用户在本机执行（沙箱策略见下）。

**可扩展性说明**：即使 MVP 不做 **GitHub PR 评论机器人**，后续也可以扩展。核心原因是：Agent 的“输入为 diff/变更集 + 上下文，输出为结构化 review/debug 结果”，而 PR 机器人只需要实现一个 GitHub 触发器/适配器，把事件转成同一套内部输入即可（Phase 2 可单独补 GitHub App + webhook/鉴权）。

### 1.3 非目标（YAGNI，避免 scope 爆炸）

- 不承诺「自动修复所有 bug 并保证通过 CI」（可定位为**辅助**，最终修改由人审阅）。  
- 第一期不强制做多智能体竞赛式架构；**单 Agent + 清晰工具集** 足够写进简历并讲清设计。  
- 不做私有代码托管；开源默认用户自带 API Key（或文档说明本地模型接入方式）。

### 1.4 成功标准（可量化）

- 公开仓库有 **README + LICENSE**，`docker compose up` 或一条命令能跑通 demo。  
- 至少 **1 条端到端用例**：给定一段有问题的代码 + 失败信息 → 产出可读的 debug 建议 + review 要点。  
- 有 **最小评测集**（10～30 个合成小任务即可）：统计格式合法率、人工 spot-check；**构造方式与公开 benchmark 用法见 §3.5**。  
- 两人协作痕迹：**Issues / PR / 清晰 commit 历史**。

## 2. 方案对比（选型思路）


| 方案                         | 描述                               | 优点          | 缺点                | 适合阶段           |
| -------------------------- | -------------------------------- | ----------- | ----------------- | -------------- |
| **A. CLI 核心**              | `review` / `debug` 子命令，读仓库与 diff | 实现快、易测、易开源  | 无 UI              | **MVP 首选**     |
| **B. FastAPI 服务**          | 对外 HTTP，异步任务队列（可选）               | 贴近岗位「服务化」叙事 | 运维与鉴权成本           | MVP+ 或与 A 并行薄层 |
| **C. GitHub Action / Bot** | PR 上自动评论                         | 展示力强        | Secret、权限、Webhook | Phase 2        |
| **D. IDE 插件**              | VS Code 内嵌                       | 体验好         | 前端/插件学习曲线陡        | 有余力再做          |


**推荐路径**：**A →（薄）B → C**。与 `final_plan.md` 中「Python + FastAPI + 可部署」主线一致。

## 3. 技术框架（逻辑架构）

### 3.1 分层（建议）

```
┌─────────────────────────────────────────────────────────────────────┐
│  入口层：CLI (Typer/Click)  ·  可选 FastAPI 路由                     │
├─────────────────────────────────────────────────────────────────────┤
│  编排层：Agent 循环（5 阶段模式，受 Claude Code 启发）                 │
│  · Phase 1: 上下文准备（按需加载相关文件/变更）                        │
│  · Phase 2: 模型分析（LLM 推理与计划制定）                           │
│  · Phase 3: 工具执行（读文件、运行测试、grep 等）                    │
│  · Phase 4: 结果处理（聚合、格式化输出、状态更新）                      │
│  · Phase 5: 继续/终止判断（是否需要进一步分析）                        │
├─────────────────────────────────────────────────────────────────────┤
│  工具层（Tool Calling，Claude Code 风格）                            │
│  · 分类工具：只读（安全/可并发）· 写入（需确认/需隔离）· 执行（沙箱执行）  │
│  · 结构化 Schema：JSONSchema/Pydantic 验证                          │
│  · 结果管理：缓存、预算控制、并发执行                                │
├─────────────────────────────────────────────────────────────────────┤
│  服务层：API 客户端 · 状态管理 · 上下文压缩 · 记忆提取                   │
├─────────────────────────────────────────────────────────────────────┤
│  模型层：OpenAI 兼容 API / 其他 Provider 抽象                         │
├─────────────────────────────────────────────────────────────────────┤
│  横切：配置、日志、结构化输出(Pydantic)、成本与 token 记录、权限管理      │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 与「Debug」强相关的工程点

- **可复现**：把模型输入输出（可脱敏）写入日志或 artifact 目录，便于复盘；实现结构化状态管理（ContextState）跟踪调试全过程。  
- **安全**：采用分级权限模式，默认不执行任意 shell；若执行命令，采取 **超时 + 工作目录限制 + 沙箱执行** 等约束，并记录运行元信息。  
- **评测**：固定若干「仓库快照 + 期望输出 schema」，避免只看单次演示；**黄金集与公开子集分工见 §3.5**。
- **状态管理**：实现 `ContextState` 结构化管理调试状态，包含目标、约束、决策历史和错误记录。

### 3.3 仓库与 Docker

- **应用镜像**：Python 3.11+，依赖 `requirements.txt` 或 `uv/poetry`（两人统一一种）。  
- **开发**：`docker-compose` 可选挂载本地代码；生产型 demo 用只读 + 环境变量注入 `OPENAI_API_KEY`。  
- **CI**：GitHub Actions 跑 lint + 单测 +（可选）镜像构建，PR 必须通过。

### 3.4 Agent 友好特性与评测设计（MVP 必做）

1. **工具 schema 化（Toolability）**
  每个工具都要有清晰输入输出（JSON/Pydantic），例如：读文件、解析 diff、搜索符号、运行测试。目标是让 Agent 在多轮调用中可预测、可复用、可回放。参考 Claude Code 的 Tool 接口模式，增加 `isEnabled()` 和 `isConcurrencySafe()` 方法。

2. **5 阶段编排模式（受 Claude Code 启发）**
  实现类似 Claude Code 的 5 阶段查询模式：
  - Phase 1: 上下文准备（按需加载相关文件/变更，实现智能上下文压缩）
  - Phase 2: 模型分析（LLM 推理与计划制定）
  - Phase 3: 工具执行（并发工具调度，智能分区）
  - Phase 4: 结果处理（聚合、格式化输出、状态更新）
  - Phase 5: 继续/终止判断（是否需要进一步分析）
  
3. **按需上下文加载（Context Budget Friendly）**
  默认只喂 `diff + 相关文件片段`，必要时再扩展到接口定义或相邻模块，避免整仓库进入上下文导致 token 膨胀与推理噪声。借鉴 Claude Code 的上下文压缩策略（microcompact, autocompact）。

4. **结构化输出（Actionable Output）**
  输出统一为可执行结构：`severity`、`location`、`evidence`、`suggestion`、`confidence`。这样既能让人快速审阅，也能直接接到后续 PR 评论或报告渲染。

5. **结构化状态管理（Context State）**
  实现 Claude Code 风格的 `ContextState` 类，跟踪调试全过程：
  ```python
  class ContextState(BaseModel):
      goal: str  # 当前调试目标
      constraints: list[str]  # 当前限制条件
      decisions: list[DecisionStep]  # 决策历史
      current_files: list[str]  # 当前关注文件
      errors: list[ErrorDetail]  # 错误记录
  ```

6. **工具并发与隔离（Streaming Tool Executor）**
  借鉴 Claude Code 的 StreamingToolExecutor 模式，实现智能工具并发执行：
  - 只读工具：安全并发执行
  - 写入工具：串行执行并需要确认
  - 执行工具：沙箱隔离执行

7. **可观测性（Observability）**
  每次运行记录 `run_id`、工具调用序列、关键中间结果、耗时和 token 成本；出现误报/漏报时能复盘”问题来自检索、推理还是工具调用”。

8. **失败可恢复（Graceful Degradation）**
  工具超时、命令失败、模型波动时要返回结构化错误，并给出降级路径（例如请求用户补充日志/手动执行命令）。保证失败时仍可输出部分有用结论。

9. **面向评测设计（Evaluability）**
  从 MVP 开始准备最小评测集（10～30 个任务），至少跟踪：输出格式合法率、关键问题命中率、误报率、人工可接受度；同时保留基线对比（例如”无规则检索” vs “规则 RAG”）。**黄金集怎么建、公开榜怎么用见 §3.5。**

### 3.5 评测方案：黄金集为主、公开 benchmark 为辅

**原则**：MVP 以 **可控、可复现、可解释** 为先，不把「刷整套公开榜」当作第一目标；公开数据仅作 **补充维度**。

#### （1）主路径：自建黄金集（Golden Set）

- **素材**：从 **小型开源仓库** 选取若干 **commit**（或等价快照），规模以「单人能在评测机/容器里拉取并跑通」为准。  
- **缺陷来源**：**人为注入 bug**，或 **保留与该 commit 对应的已知 issue / 失败用例**（二选一或混用，每条任务在评测清单里写清）。  
- **固定输入**：为每条任务固化 **diff / 相关文件片段 / 错误日志或测试失败输出**（与产品真实输入形态一致），存入版本库或 artifact（如 `eval/fixtures/`），避免「每次手敲 prompt」。  
- **期望行为（可自动或半自动判分）**：在任务元数据中写明，例如：  
  - **检出类**：输出至少命中某一 **问题类别**（如空指针风险、错误异常处理、某条安全模式）；  
  - **结构类**：结构化输出 **必须包含某字段**（如 `severity`、`location`）或通过 JSON schema 校验；  
  - **定位类**（若启用）：是否指向正确 **文件路径** 或 **行号范围**（允许宽松匹配规则，并在文档中写死）。  
- **指标**：与 §3.4 一致——格式合法率、关键问题命中率、误报率、人工 spot-check；必要时加 **耗时与 token** 作为工程回归指标。
- **可复现性**：参考 Claude Code 的实践，确保评测输入输出可脱敏记录，支持复盘和调试。

#### （2）补充维度：公开 benchmark（不整套绑死）

- **定位**：在黄金集跑通后，用 **小样本子集** 做 **外推与叙事**，例如从 **SWE-bench** 等数据中 **抽取少量实例**，且 **只评与产品一致的目标**（如「能否定位到相关文件」「是否提出合理假设」），**不**默认承担「端到端修 patch 并合并」的完整榜单定义。  
- **约束**：子集规模、筛选规则、模型与工具版本 **写进评测说明**，避免「挑简单题」或不可复现；与主评测的 **通过/失败口径** 分开汇报，避免混为一谈。
- **基准对比**：建立类似 Claude Code 中的对照实验，比较不同架构选择的效果差异。

#### （3）与 CI / 迭代的关系

- 黄金集可随 **评测脚本** 在 CI 中跑 **轻量子集**（或仅 schema 校验），全量评测可在发布前或定期手动触发。  
- 新增 Agent 能力时，优先 **追加黄金集任务** 再考虑扩充公开子集。
- 参考 Claude Code 的实践，对关键变更进行回归测试。

## 4. 开发流程（两人远程）

### 4.1 Git / GitHub

- **主分支**：`main`，保护分支，仅通过 PR 合并。  
- **分支命名**：`feat/…`、`fix/…`、`chore/…`。  
- **PR 要求**：描述动机、截图或示例输出、关联 Issue；互相 review 再合并。  
- **Issue**：用标签区分 `mvp`、`phase-2`、`bug`、`docs`。

### 4.2 迭代节奏（建议 1～2 周一里程碑）


| 周次  | 里程碑                                        |
| --- | ------------------------------------------ |
| W1  | 仓库骨架、CLI 能调模型、1 个「纯 review」路径跑通            |
| W2  | 接入读文件 + diff；debug 路径 prototype；Dockerfile |
| W3  | 工具执行策略 + 评测集 v0；README 与 LICENSE           |
| W4+ | FastAPI 薄层或 GitHub 集成（按精力二选一）              |


### 4.3 沟通

- 每周固定 1 次短会 + 异步文档（本文件与 Issue）。  
- 重大设计变更先改本文档再写代码。

## 5. 初步分工（功能模块型，可按兴趣对调）

> 原则：**接口先行**（例如 `tools/` 的函数签名与 JSON schema），避免两人长期改同一文件冲突。**深度参与机制**：双人结对编程（模块接口处）、模块互换体验、联合调试、共同测试。

### 5.1 功能模块划分

| 角色 | 功能域 | 主要职责 | 交付物示例 | 完整功能闭环 |
|------|--------|----------|------------|--------------|
| **Analyzer Agent** | 核心分析引擎 | 代码分析、上下文管理、结构化输出、推理引擎、状态管理 | `src/analyzer/`, `src/context_state.py`, `src/output_formatter.py`, `src/inference_engine.py` | 实现从代码输入到分析结果的完整链路 |
| **Integration Agent** | 工具与交互层 | CLI交互、工具系统、权限安全、外部集成、部署配置 | `cli.py`, `src/tools/`, `src/security/`, `src/integrations/`, `Dockerfile`, `.github/workflows/` | 实现从用户输入到工具执行的完整链路 |

### 5.2 共同开发区域

**交叉合作领域**（需共同设计与维护）：
- **Prompt 模板设计**：共同讨论系统提示词、分析指令模板，确保逻辑一致性与工程可用性
- **JSON Schema 定义**：共同设计输出格式，包括错误定位、代码审查建议、调试步骤等结构化输出
- **工具接口定义**：Analyzer Agent 设计功能逻辑，Integration Agent 设计接口规范和安全边界
- **API 协议定义**：共同设计 Analyzer 与 Integration 之间的数据交换协议
- **测试策略制定**：共同规划单元测试、集成测试、端到端测试覆盖策略

### 5.3 深度参与机制

- **双人结对编程**：在核心模块接口处进行结对编程，确保模块间无缝对接
- **模块互换体验**：在开发过程中短暂交换模块开发体验，加深对整体架构的理解
- **联合调试**：共同调试模块间的集成问题，提升协作效率
- **共同测试**：Analyzer Agent 负责 Integration Agent 的功能测试，Integration Agent 负责 Analyzer Agent 的集成测试
- **周度同步会**：每周固定时间同步开发进度，共同review关键代码

## 6. 待确认清单
